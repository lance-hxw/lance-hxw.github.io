### 1. 双端List（Double-Ended List）

双端列表是一种支持在两端快速插入和删除元素的数据结构。它的特点是：

- **操作效率**：支持在头部和尾部进行常数时间复杂度（O(1)）的插入和删除操作。
- **内存使用**：在Redis中，双端列表是通过链表实现的，每个节点存储元素和指向前后节点的指针，因此在内存的使用上可能较为低效，尤其是在元素较少时，因为每个元素都需要额外的存储空间来维护指针。

### 2. QuickList

QuickList 是 Redis 内部实现的优化列表结构，结合了双端链表和压缩列表的优点。

- **基本原理**：QuickList 是由多个压缩列表（ZipList）构成的双端链表。每个节点存储一个压缩列表，而这些压缩列表连接在一起，形成一个双端链表。
- **操作效率**：在 QuickList 中，元素的插入和删除操作依然是 O(1)，但它利用压缩列表减少了内存的占用。压缩列表对于小型数据的存储特别高效。
- **内存优化**：由于压缩列表的使用，QuickList 能在处理较小的数据时比单纯的链表节省更多内存，并且在处理大数据时仍然保持较好的性能。
- **优势**：在 Redis 3.2 以后，Redis 默认将所有的列表实现为 QuickList，因此它在很多场景下被广泛使用，尤其是在需要高效内存利用和快速数据访问的场景中。

### 3. ZipList

ZipList 是一种为小型数据结构优化的内存节省型数据结构，通常用于存储较短的列表。

- **基本原理**：ZipList 是一种扁平化的连续内存块，它通过将多个小元素压缩成一个内存块来节省空间。每个元素的长度和内容都会被压缩，以减少内存的使用。
- **操作效率**：对于大多数操作，ZipList 都能保持较低的内存开销，但它的缺点是在需要频繁插入、删除元素时性能较差，因为它需要重新组织内存块。
- **使用场景**：ZipList 最适合用于存储短小的数据（例如，长度小于 512 字节的列表），并且仅适用于较少的操作。Redis 在默认情况下对于某些较小的列表会使用 ZipList 来优化内存。

#### 具体实现: 将字符串/整数压缩成一块, 用字节长度记录每个节点内容的大小
- 头结点记录尾巴的偏移量, 总元素数量, 结束点(固定值)等
- 每个节点记录当前节点类型和长度, 实际数据
	- 还记录前一个节点长度, 用于倒序索引
- 显然这东西只能存小列表
- 最大优点是, 通过不同长度的数据设计, 极度节省内存
#### 重大缺点:
这东西内存上是一块的, 一旦元素变动(变动), 或者插入元素, 就会导致内存分配
关键是有连锁更新这个东西: 只要改变一个元素, 所有的后续元素都会需要改动, 直接反复波动.


### 4. ListPack

ListPack 是 Redis 在 5.0 版本中引入的一种新的高效内存压缩数据结构，主要用于优化小型列表的存储。

- **基本原理**：ListPack 是一个更加高效的内存压缩技术，它将多个元素打包在一个连续的内存区域中，以减少内存的碎片化，并通过更高效的编码方式进一步压缩数据。
- **操作效率**：在对小数据量的访问时，ListPack 提供比 ZipList 更好的性能和更高的压缩率。其插入、删除操作的时间复杂度是 O(1)，而且它不需要像 ZipList 那样进行内存重排，因此它在处理小型列表时更加高效。
- **内存优化**：由于采用了更加高效的压缩算法，ListPack 在内存占用上相比 ZipList 可能有更大的优势，尤其是在包含大量小元素时。
- **使用场景**：ListPack 主要用于存储长度较短的数据，如常见的短列表、短时间数据等。
### 相比ziplist 改进： 不再存储前节点长度， 避免了连锁更新， 只存放当前节点结束位置
### 总结

- **双端List**：适用于需要频繁在两端进行操作的大型列表，内存消耗较大。
- **QuickList**：当前Redis的默认实现，结合了链表和压缩列表的优点，适用于大多数场景。
- **ZipList**：专为存储小型数据设计，内存使用高效，但不适合频繁修改的场景。
- **ListPack**：提供比ZipList更好的内存优化，适合存储较短的列表和高效的内存管理。
