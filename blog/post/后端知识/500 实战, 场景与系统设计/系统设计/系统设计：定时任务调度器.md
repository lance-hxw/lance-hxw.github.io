场景：
有很多任务，每个任务有时间戳，是预计开始的时间

# 实现1: 阻塞优先队列+polling
就用juc的阻塞优先队列，线程安全，非常方便
手动实现就需要用优先队列+锁去实现
但是轮询不好，主要是时间间隔设置问题

# （不可用）实现2: 阻塞优先队列+时间差
不再固定时间去询问，而是每次查询的时候，记录最上面没有到期的任务的时间戳，然后sleep到那个时候

看似优美，实际不可用。。
- 如果在等待队头期间有紧急任务放进来，根本处理不了
# DelayQueue
java的实现，他将方案2的功能集成进去并解决了“队头睡觉”问题
使用者直接不断take就行了，会自动睡觉

解决上述问题的方案是，如果放到头上去就唤醒等待的消费者
- 实现逻辑是， 插入后，检测队头是不是就是刚才插入的元素，如果是， 就唤醒

特殊设计：有leader线程，如果leader非空， 说明已经有消费者访问过了，并且正在阻塞等待队头到期

# 时间轮
即循环队列
如一个8位的循环队列，将所有任务的时间取模后放到对因的队头后面
记录剩余圈数和位置
按位置访问，每次访问圈数-1
如果为0， 就执行
这个方法的插入和删除的复杂度都很低

netty和linux内核中的定时器都是这种设计


# 分布式定时任务调度？
redis zset
- 将时间作为score，然后轮询
redis keyspace notificatons
- key过期时触发事件通知
- 程序订阅这个过期事件即可，解析键名执行相关业务
- 不需要轮询，但redis支持的过期时间支持优先
rabbitmq
- 使用延迟消息插件实现延迟消息功能
- 任务发送到队列，延迟时间到达后才会被接受进队列
- 然后正常消费就行了
- 这是天然解耦的设计，但是需要加一个插件，需要配置和维护
