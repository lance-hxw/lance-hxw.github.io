消息推送， 个人理解蕴含了：
- feeds， timeline或者推荐系统
- 通知，数据只存客户端
- 私信，即时性要求高

其中，后两者主要侧重于即时消息推送
# 写入

[[公共信箱+个人信箱]]
这里主要关注公共信箱和个人信箱， 是fanout的时机， 是信息存储和组织方式

- 公共信箱， 是把msg放在公共池， 读的时候去过滤组装
- 个人信箱， 是在msg生成时复制到各自的信箱

# 通知(消费/获取)

[[系统设计：feed流（推文系统）]]
是client获取msg的方式
- push
	- 用长连接，移动端通知（APNS/FCM），MQTT等主动推送
	- 优点：实时性强
	- 缺点：可靠性实现复杂，消息量大时系统压力大
- pull
	- client通过接口主动拉
	- 优点：实现简单，没有复杂逻辑，不需要长连接开销， 压力小
	- 缺点：实时性不行， 容易有无效开销


# 生产与消费并非耦合

- fanout on write 似乎天生与push匹配
	- 准备好了每个人的msg， 可以直接送
	- 这种一般是IM场景， 直接把msg全文推出去
- fanout on read 似乎天生与pull匹配
	- 公共池子中的msg在pull时过滤和计算
	- 这种一般是推荐流，在api拉的时候计算

但是也可以：

- 公共邮箱+push
	- 对于需要广播的内容， 可以push“有新内容了”， 然后让client去pull
	- 这个push一般是长连接去做
- 个人邮箱+pull
	- 这个是定时拉自己的inbox， 不过这个不太好， 效率不好
